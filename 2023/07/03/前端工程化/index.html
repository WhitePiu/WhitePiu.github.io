Node.js

Node运行js程序的底层原理：

JS代码首先经过V8引擎，通过Node中的Bingdings，将任务放到Libuv的事件循环中

libuv是用C语言编写的库

libuv提供了事件循环、文件系统读写、网络IO、线程池内容，处理完的事件放入事件队列再返回V8，JS中进行回调

Node应用场景

  • 前端开发库以node包进行管理
  • npm,yam,pnpm工具为前端开发
  • web服务器开发，中间件，代理服务器
  • 前后端渲染同构应用
  • 编写脚本工具，Node运行
  • Electron开发桌面应用程序

Node 版本控制工具：nvm-windows

电脑上可以安装多个Node版本并能根据需要灵活切换，官方的两个控制工具n、nvm都不支持Windows。github上有开发出来的nvm-windows，我们用这个进行控制

nvm list表示可以控制的版本

nvm use直接切换node版本号

Node输入

在终端用Node执行JS代码后面可以跟参数，在JS中process.argv[]可以获取某一个参数，与在Linux中的Shell编程同理

Node中一些特殊的全局变量

Node中没有Window对象，但有个类似的global对象

  • __dirname：当前文件所在的目录结构
  • __filename：当前目录+文件名称
  • process：有关当前进程的信息
  • 各种各样的定时器方法，浏览器的那一套Node也有

模块化部分

模块化

将程序划分为一个一个小的结构，并且互相不影响，有助于开发

在早期JS是没有自己的模块化编程方案的，社区就自己搞出了一套很流行的方案：commenJS

CommenJS

commenJS规定模块之间的沟通必须要经过传出数据模块的导出，传入数据模块的导入，但是他是社区的，浏览器不支持CommenJS所以在JS编写CommenJS由html引入JS文件是运行不了的，但是Node，webpack等内部实现了CommenJS，支持CommenJS的开发，所以我们用Node来进行学习测试

本质：exports是一个对象，导入导出就是将exports引用赋值给util

module.exports导出：module对象的exports属性是上一个exports的指针，所以也是同一个对象，但是Node是通过操作module.exports导入导出的，一旦module.exports写成新建的对象的形式，就不是同一个对象了，exports就没用了所以开发一般用module.exports

require的查找细则：

  1. 根据路径导入自己编写的模块，没有后缀名会按X,X.js,X.json,X.node顺序查找，都没有就当成一个目录找这个目录下的index，index.js，index.json，index.node文件
  2. 导入node提供的内置模块：require（”内置模块名称”），如果没有，那就不停在上层目录的node_modules（名字固定）目录下找，一直找到根目录

require会自动执行被导入JS文件一次

多个模块的加载顺序：多个模块与模块之间的关系就是一个有向图！！加载的顺序是DFS的顺序

ES6推出自己的模块化方案ESmodule

ESmodule

export关键字导出

import关键字导入

vue中已经使用过就不细说了

  • as关键字可以起别名，在导入导出都可以起别名，类似数据库sql语句
  • 也可以在定义变量前直接加export导出

default用法：

当只导出一个重要数据时，且文件名已经能代表用途，就用default默认导出（一个模块只能有一个默认导出）

包管理工具

npm

安装：属于Node自己的管理工具，安装Node自动安装npm

npm下载包会直接下载到当前目录的node_modules文件夹下

npm install dayjs dayjs包就会出现在当前文件夹下

package.json

用于记录各种各样的项目信息

手动从零创建项目 npm init -y

常见字段解释：

  1. name：项目名称
  2. version：当前项目版本号
  3. description：项目描述信息
  4. author：作者相关信息
  5. license：开源协议
  6. private：true代表私有，npm不能发布
  7. main：设置程序入口（上面谈到require会自动匹配文件夹下的index.js，但是如果有package.json，node会先解析package.json文件，如果main入口配置的是*.js，那么默认匹配就会变成*.js）
  8. scripts：是一个对象，里面放键值对
    • start：加一个node 的命令，相当于有一个直接运行的脚本
    • build: 打包

终端npm run start就可以运行 node ./abc.js

  1. dependencies：项目的依赖，运行需要的包也是一个对象，编写键值对

npm install 将自动下载dependencies内的包

  1. devDependencies：开发依赖，生产环境用不到的会放在这里（都用到就是dependencies这里）

npm install xxx --save-dev/-D 就会放到devDenpendencies

  1. peerDependencies：依赖的依赖，就是你依赖的一个包，必须以另一个包为前提

开发依赖：理解为开发者必须要用的，webpack等

生产依赖：使用者必须要用到，vue等

npm install

npm install xxx：安装到当前文件夹

  • -g：全局安装，安装到Node专门管理的目录，且会添加到环境变量，一般安装工具型的包
  • -D：只是开发依赖

npm 缓存：npm会把一些下载的包放入缓存，防止多次下载一个包每次去官网registry中下载，而每个包的标识符信息就会记录到package-lock.json中，经过算法找到对应的缓存应该是计算机组成原理的知识应用

package-lock.json文件：

  • lockfileVersion：lock文件版本
  • requires：使用requires来跟踪模块的依赖关系
  • resolved：记录下载地址，在registry仓库中的位置
  • integrity：缓存标识符

这个文件操作全部自动，一般不用手动维护，能查看包与包之间明确的依赖信息，也能看到具体下载的什么版本的包

npm其他命令：

  • npm uninstall xxx：卸载
  • npm rebuile：重新build
  • npm cache clean：清除缓存

yarn

为弥补早期npm速度慢，版本混乱的缺陷而出现，等npm更新了以后基本相似没区别

yarn init -y 初始化package.json配置

yarn add xxx添加包

cnpm

npm在国内的镜像服务器

pnpm

越来越流行的管理工具，原来的包管理工具有一个这样的问题：如果有多个项目用的同样的库，那么这个库也会存在多份，大大占据了储存空间，pnpm的出现就是解决这个痛点

pnpm多少个项目，同一个依赖包在磁盘上只有一份存储，不同项目是通过硬链接实现调用包

pnpm采用非扁平化管理：

扁平化的含义：按理说比如我安装webpack，里面有许多依赖包，按逻辑应该安装在webpack下的node_module文件夹下，扁平化的意思是把所有包都放到项目文件夹下的那个唯一的node_module

Webpack

node内置path模块

不同操作系统表示文件的方式不同：Windows：xxx\xxx.txt，Linux：xxx/xxx.txt

path模块就是对不同的操作系统生成对应的路径

版本控制工具：Git

简单来说就是游戏存档啊

集中式版本控制：早期的版本控制，所有的数据备份都在一个服务器端

分布式版本控制：每台电脑都能把代码仓库完整的镜像下来，git就是分布式

  1. 每台电脑安装git后第一步就是配置信息

配置名字： git config --global user.name "xxx"

配置邮箱： git config --global user.email "xxx@xx.com"

  1. 正式管理项目，创建本地仓库，就在当前项目文件夹下

git init

  1. 将项目的所有文件加入git的管理中

git add .

  1. 把远程仓库完整的下载下来

git clone "xxx(地址)"

  1. 提交到git仓库

git commit -m "（简介）"

  1. 查看文件的状态

git status

  1. 查看修改历史

git log

  1. 版本回退

git reset --hard HEAD^ 回退到上一个版本

git reset --hard HEAD~100 回到上100个版本

git reset --hard xxx(commit id) 回到对应哈希值的版本

  1. 上传远程仓库

git push

对于在git管理文件中的状态

  • 未跟踪：默认创建的文件不会直接添加到git仓库中，要用git add命令
  • 已跟踪：添加到git仓库中的文件就是已跟踪状态

已跟踪文件划分：

  • staged：暂缓区中的文件状态
  • Unmodified：commit命令，此时将staged文件可以提交到git仓库
  • Modified：修改了某个文件进入的状态

.gitignore

列出git的忽略管理的文件

git的服务器操作

私有的远程仓库当然不能随意的访问，需要凭证，但是http协议是无状态连接（就是每次登录都要账号密码），操作麻烦，但是Git拥有一个凭证系统来处理这个问题

对于Windows系统有类似于密钥的辅助工具 Git Credential Manager for Windows可以解决这个问题

SSH密钥

SSH是加密的网络传输协 议，可以在不安全的网络中为网络服务提供安全的传输环境

非对称加密的形式：自己客户端的电脑要生成一对公钥和私钥，公钥放在服务器端，私钥放在客户端，每次访问远程仓库就会自动匹配两台机子的密钥，实现登录连接

ssh-keygen -t ed25513 -C "xxx(邮箱)"

找到生成的公钥，配置到对应的git服务器上（比如Github），就完成了

与远程仓库建立连接

git remote add origin（别名） "(地址)"

从远程仓库拿来代码

git pull origin master(具体哪个分支)

设置上游/跟踪分支,在此之后就能直接打 git pull了

git branch --set-upstream-to=origin/master

分支

git的本质：每一次commit会形成一个对象，这个对象指向另一个记录着这次提交关联的文件的对象，而第二个的这个对象里面全是二进制文件的指针，二进制文件就是我们的源代码文件，因此git可以存下来多个版本。

进一步来说，git每一次提交master指针都向后移一次，每一个提交对象里都有连向上一次提交的的指针，也就是说，如果我们就是一步一步开发的话，整个项目仓库的版本互相联系又能互相到达，根本原因其实是他内部这样形成了一个链表结构，什么tag标签啊其实就是新建了一个不会随便乱动的指针，指向某一个commit对象罢了

在进一步来说，后面所要学习的git分支，就很容易想到应该就是在某一个结点连向了多个儿子，变成了一颗树的结构，指针是双亲指针，在数据结构算法角度考虑那么一个git仓库就是一个并查集！！！

master分支：就是指向当前最后一次的提交

HEAD指向谁当前分支就是哪，HEAD默认指向master

创建新的分支，其实就是创建了一个类似于master的可移动的指针

git branch text 创建分支

git checkout text切换分支

git merge hotfix 合并分支（能合并前提是必须有共同祖先），分支合并后会有两个父亲指针，就会变成图结构

git flow

一般大公司的开发规范

一般在develop分支进行主要开发

master一般合并develop构成一些很稳定的版本

feature分支一般开发新功能

hotfix分支一般用作修复bug

git rebase

因为merge完了以后这块分支的结构就不是线性的了，rebase就是能改变父亲指针，然后合并之后就能改变成线性结构（开发不常用，主分支禁用），看效果比较危险，不是很熟练的开发者就先别用

git rebase master base改成master

git checkout master 切换到master

git merge feature 最后再合并
